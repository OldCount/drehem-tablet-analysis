## Overview

Addressed a batch of bugs reported from manual tablet review across five test tablets
(P200031, P332109, P128123, P123620, P127531). Changes span the extraction engine

## drehem_extract.py

### New vocabulary entries
- `MEASURE_TERMS`: added `balag` (ritual harp/drum), `ma2` (boat), `kaskal`
  (journey/expedition)
- `NON_ANIMAL_LINE_MARKERS`: added `siskur2 a-sza3-ga` (field offering) and `a-sza3-ga`
  (field/meadow context)
- `ANIMAL_QUALIFIERS`: added `nu2-a` (mating/lying down), attested in P128123 line 1
- `DESTINATION_CATEGORIES`: added `tum-ma-al` (Tummal, as sacred-site) and `e2-gal-la`
  (palace variant of e2-gal)

### Destination extraction â€” extract_destination()
- Added `sza3 PLACE` pattern: recognises "in PLACE" constructions, e.g.
  `sza3 tum-ma-al` in P127531
- Added `DEST ba-an-kux` pattern: recognises palace-delivery constructions such as
  `e2-gal-la ba-an-kux` in P123620

### Notation
- Added comment to `MEASURE_TERMS` clarifying that `ga` is retained there for
  dairy contexts; disambiguation is handled in the visualizer layer (see below)

## tablet_visualizer.py

### Translations
Added English and Dutch translations to `TRANSLATIONS` for all newly recognised terms:
`balag`, `ma2`, `siskur2`, `a-sza3-ga`, `nu2-a`, `kaskal`, `sza3`, `bala`.

### ga context disambiguation
When `ga` appears on a line that contains an extracted animal entry, it is now tagged
`qualifier / suckling` rather than `measure / non-animal commodity`.

### ba-ba-ti maszkim â€” partial fix
A `person_part` role was added: if a token is a hyphen-component of a known multi-part
person name that appears on the same line, it is tagged as `person_part` to prevent
fall-through to transaction-keyword matching. The deeper tokenizer issue (matching
`ba-ti` inside `ba-ba-ti`) may require further targeted testing.

### Divine recipient translations
The annotation JSON now includes a `divine_recipients_translated` field with `name`,
`en`, and `nl` sub-fields for each deity. The frontend has not yet been updated to
render these translations; the data is present in the API response but unused in UI.

## Remaining Issues (not addressed this session)

- `[...]` tokens are not yet consistently tagged as `damage`; some fall through to
  role-marker classification
- P128123: `szesz-da-da` (person), `nibru{ki}` (city of Nippur), and `na-sa6` (person
  Nasa) are not extracted by the current person/location parser
- Komma notation `s,` (representing the sibilant ts/s,) is not normalised before
  name matching; affects e.g. `s,e-lu-usz-{d}da-gan` in P128123
- P200031 line 5: `la2` subtraction in date formula (`3(u) la2 1` = day 29) is parsed
  as the raw value rather than the computed result
- `iti-ta ... ba-ra-zal` date formula (P123620 line 3: "of the month, day N has
  passed") is not recognised by the date extractor
- Frontend: `divine_recipients_translated` data is present in API output but the panel
  UI has not been updated to render the translations
